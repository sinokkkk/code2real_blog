本次中期项目使用 ESP32-S2 Mini、128x64 SSD1306 OLED、DHT11 温湿度传感器和按键，完成一个小型多功能桌面屏。项目不是单独验证某一个模块，而是把时钟、温湿度、系统信息、按键菜单和 Bad Apple 串口播放整合到同一个工程中。

主工程文件为：

```text
/esp32_s2mini_oled_clock_modules/esp32_s2mini_oled_clock_modules.ino
```

项目当前已经具备完整演示能力：开机后初始化 OLED，连接 WiFi 同步时间，再通过按键在不同页面之间切换。进入 Bad Apple 页面时，ESP32 会向电脑端发送播放命令，由电脑端脚本通过串口推送视频帧。

<img :src="'/code2real_blog/posts/mid/1.jpg'"  />

## 项目目标

项目初始目标是完成一个“多功能桌面信息显示系统”。OLED 屏幕负责显示页面内容，ESP32-S2 Mini 负责网络时间、传感器读取、按键交互和串口通信。

核心功能包括：

- 使用 OLED 显示基础信息和动态图像。
- 使用 WiFi 获取网络时间，实现时钟页面。
- 使用 DHT11 采集温湿度。
- 使用一个按键完成页面切换和菜单选择。
- 通过电脑端串口推流，实现 Bad Apple 在 OLED 上播放。

这个项目的重点不在于某个模块单独跑通，而在于多个功能并存时的资源协调。引脚分配、通信稳定性、页面刷新和交互逻辑都会互相影响。

## 硬件方案

硬件部分主要由 ESP32-S2 Mini、SSD1306 OLED、DHT11 和一个按键组成。OLED 使用 I2C 通信，DHT11 使用单总线通信，按键使用内部上拉输入，USB 串口用于和电脑端 Bad Apple 推流程序通信。

当前主要引脚分配如下：

```text
OLED SDA   -> GPIO33
OLED SCL   -> GPIO35
Button     -> GPIO18
DHT11 DATA -> GPIO7
```

前期曾尝试使用 GPIO4 作为 DHT11 数据脚，但在 WiFi 和传感器同时使用时表现不稳定。后续将 DHT11 迁移到 GPIO7，避开 OLED、按钮和 USB 相关引脚后，系统稳定性明显改善。

这一步让我意识到，嵌入式项目里的“可用 GPIO”并不等于“随便用”。有些引脚可能和启动、USB、板载功能或外设存在隐性冲突，只看代码无法判断，必须结合实际板卡测试。

## 软件架构

软件采用多页面状态机结构。程序中定义了 `ScreenId` 枚举，用于表示当前显示页面：

```cpp
enum ScreenId {
  SCREEN_CLOCK = 0,
  SCREEN_TEMP,
  SCREEN_INFO,
  SCREEN_BADAPPLE,
  SCREEN_MENU,
  SCREEN_COUNT
};
```

主循环主要做三件事：

- `updateButton()`：处理按键输入。
- `updateBadAppleStream()`：处理串口视频流。
- `drawCurrentScreen()`：刷新当前页面。

目前页面包括时钟、温湿度、系统信息、Bad Apple 和菜单。这样的结构比较轻量，没有引入 RTOS 任务划分，适合当前项目规模。每个页面对应独立绘制函数，后续增加页面时只需要扩展枚举和显示逻辑。

中期阶段的关键不是把多个示例代码拼在一起，而是让它们在同一个工程里稳定切换。否则单个模块能运行，合并后仍然可能因为刷新频率、串口占用或引脚冲突出现问题。

## OLED 显示模块

OLED 使用 SSD1306 驱动芯片，分辨率为 128x64。程序使用 `Adafruit_SSD1306` 和 `Adafruit_GFX` 完成绘制。普通页面以文字为主，例如 Clock 页面显示时间和日期，Info 页面显示堆内存和当前页面信息。

OLED 初始化代码如下：

```cpp
Wire.begin(OLED_SDA, OLED_SCL);
display.begin(SSD1306_SWITCHCAPVCC, OLED_ADDR);
```

为了让页面风格统一，程序中设计了 `drawHeader()`，在每个页面顶部绘制标题和分隔线。这样不同页面虽然内容不同，但视觉结构保持一致。

Bad Apple 页面采用另一种刷新方式。它不逐个绘制文字，而是直接把电脑端发送的 1024 字节帧数据复制到 SSD1306 显存缓冲区：

```cpp
memcpy(display.getBuffer(), badAppleFrameBuffer, FRAME_BYTES);
display.display();
```

这种方式更适合黑白视频帧播放，也减少了 ESP32 端的图像处理压力。



## WiFi 时钟模块

时钟模块通过 WiFi 连接网络，再使用 NTP 服务器同步时间。代码中使用了以下服务器：

```cpp
configTzTime(TZ_INFO, "ntp.aliyun.com", "ntp.tencent.com", "pool.ntp.org");
```

时区设置为：

```cpp
static const char *TZ_INFO = "CST-8";
```

WiFi 连接成功后，系统获取本地时间，并在 Clock 页面显示小时、分钟、秒和日期。为了避免 WiFi 长时间占用资源，NTP 同步完成后会主动断开 WiFi 并关闭 WiFi 模块。

调试时遇到过 WiFi 偶尔连接失败的问题。最初程序只调用 `WiFi.begin()`，如果上一次连接状态没有清理干净，可能等待很久仍无法连接。后续在连接前加入 `WiFi.disconnect(true)`，关闭 WiFi 省电模式，并将等待时间延长到 25 秒，连接过程稳定了很多。

## DHT11 温湿度模块

温湿度模块使用 DHT11。DHT11 采用单总线通信，只有一根 DATA 线。主机先将 DATA 拉低约 20ms，再释放总线，等待传感器响应，随后读取 40bit 数据。

DHT11 返回的数据包括：

```text
湿度整数部分
湿度小数部分
温度整数部分
温度小数/符号部分
校验字节
```

程序中定义结构体保存读取结果：

```cpp
struct Dht11Reading {
  float temperatureC;
  float humidityRh;
  bool valid;
};
```

读取时会计算校验和。如果校验失败，程序不会更新有效数据显示，避免 OLED 上出现错误温湿度。

调试中最明显的问题是 DHT11 一直显示无数据。开始怀疑是时序代码问题，后来发现 GPIO4 作为数据脚时系统不稳定。更换到 GPIO7 后，读取成功率明显提高。为了提高可靠性，读取过程中使用 `micros()` 计时，并在关键读取阶段短暂关闭中断，减少微秒级时序被打断的概率。

## 按键交互

项目只使用一个按键完成页面切换。按键接在 GPIO18，采用 `INPUT_PULLUP` 模式，未按下时为高电平，按下时为低电平。

交互逻辑如下：

```text
短按：切换页面或移动菜单选项
长按：进入菜单或确认菜单选项
```

程序中对按键做了消抖处理，避免机械抖动导致一次按下被识别成多次点击。相关变量包括：

```cpp
buttonStableState
buttonLastReading
buttonLastDebounceMs
buttonPressStartMs
buttonLongPressHandled
```

一个按键控制多个页面的优点是硬件简单，缺点是操作效率不如多个按钮直观。因此菜单页面加入了提示文字：

```text
Click: next  Hold: enter
```

目前交互已经能满足演示需求，后续还可以继续调整长按时间、页面顺序和菜单提示。

## Bad Apple 串口播放

Bad Apple 模块采用电脑端实时转码、ESP32 端实时接收并显示的方案。ESP32-S2 Mini 存储空间有限，如果直接把完整视频放进板子会比较麻烦，因此中期阶段选择 USB 串口流式播放。

电脑端 Python 程序主要完成以下工作：

- 调用 `ffmpeg` 读取视频。
- 将视频缩放为 128x64。
- 转为灰度帧。
- 根据阈值压缩成 SSD1306 的 1-bit 显存格式。
- 每帧前加 `BA` 帧头，通过串口发送给 ESP32。

每一帧大小为：

```text
128 * 64 / 8 = 1024 字节
```

ESP32 端检测到 `BA` 帧头后，读取完整的 1024 字节帧数据，然后刷新 OLED 显存。

控制方式是 ESP32 向电脑端发送命令：

```text
CMD:START_BADAPPLE
CMD:STOP_BADAPPLE
```

电脑端守护脚本收到命令后开始或停止推流。调试中遇到过串口被占用的问题，表现为 Python 报错 `PermissionError: 拒绝访问 COM7`。通常原因是 Arduino 串口监视器或其他程序占用了串口。关闭串口监视器后，再启动 Python 守护脚本即可恢复。

## 主要问题与处理

中期开发中遇到的问题主要集中在四类。

第一是引脚冲突。最开始 DHT11 使用 GPIO4，但 WiFi 或传感器读取不稳定。处理方式是重新规划引脚，将 DHT11 DATA 移到 GPIO7。

第二是 DHT11 时序。DHT11 的高低电平持续时间只有几十微秒，普通读取方式容易超时。处理方式是使用 `micros()` 进行脉冲计时，放宽超时阈值，并保留串口错误输出，便于判断是无响应、低电平超时、高电平超时还是校验失败。

第三是 WiFi 连接。初期 WiFi 失败时只能在 OLED 上看到失败提示，很难判断具体原因。后续加入串口状态码输出，并在连接前清理旧连接状态，连接后关闭 WiFi，减少对其他模块的影响。

第四是 Arduino 编译重复定义。调试时目录中出现过复制出来的 `.ino` 文件，Arduino IDE 会把同一目录下所有 `.ino` 一起编译，导致变量和函数重复定义。最终只保留主工程 `.ino` 文件，移走多余备份文件。

这些问题说明，嵌入式调试不只是改代码。硬件连线、开发环境、串口占用和文件组织都会影响最终结果。

截至中期阶段，项目已经完成以下内容：

- Temp 页面读取并显示 DHT11 温湿度。
- Info 页面显示系统状态和当前引脚信息。
- Menu 页面支持按键选择。
- Bad Apple 页面支持电脑端串口推流播放。
- 按键短按、长按逻辑基本完成。
- WiFi 和 DHT11 的主要冲突已通过引脚调整和连接逻辑优化解决。

当前项目已经能作为中期展示使用。它不是多个独立示例的简单堆叠，而是一个能够切换页面、读取传感器、同步网络时间并播放串口视频的小系统。



## 总结

中期阶段已经完成从单个示例到多模块系统的整合。项目实现了 OLED 显示、WiFi 时钟、DHT11 温湿度、按键菜单和 Bad Apple 播放，也处理了引脚冲突、串口占用、DHT11 时序不稳定、Arduino 多 `.ino` 文件重复编译等问题。

目前系统结构已经比较清楚。主工程集中在 `esp32_s2mini_oled_clock_modules.ino` 中，各模块通过函数划分，页面通过状态机切换。后续重点会从“能运行”转向“更稳定、更易用、更适合展示”。
